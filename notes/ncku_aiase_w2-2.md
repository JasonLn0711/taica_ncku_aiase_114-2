# 📖 課程講義：規格驅動開發與產品需求定義 (PRD & SDD in Vibe Coding)

## 一、 核心概念定義：PRD 與 SDD 的分工與定位

在 AI 協作開發中，最忌諱給予 AI 模糊、發散的指令。我們必須透過標準化的工程文件，在不同維度上對 AI 進行「約束 (Constraints)」。

### 1. 產品需求文件 (PRD - Product Requirements Document)

* **定義**：PRD 定義了產品的 **「目標與範圍 (What to do)」**。它是從使用者視角出發，描述系統應該解決什麼問題、具備哪些功能，以及驗收標準。
* **核心要素 (以課程 `stockcheck.py` 為例)**：
  * **問題陳述與目標**：解決投資人需逐一開啟網頁看盤的痛點，建立批次查詢 CLI 工具。
  * **功能需求 (Functional Requirements, FR)**：例如 `FR-1` 規定輸入檔必須是嚴格的 `SYMBOL:EXCHANGE` 格式；系統需具備查詢 Google Finance 的能力。
  * **非功能需求 (Non-Functional Requirements, NFR)**：不涉及具體業務邏輯，但關乎系統品質。例如要求使用單一 `stockcheck.py` 腳本、查詢間隔需設定 0.5~1 秒（避免被封鎖）、定義明確的 Exit Code（0=成功，1=部分失敗，2=嚴重錯誤）。



### 2. 軟體設計文件 (SDD - Software Design Document)

* **定義**：SDD 定義了工程實作的 **「方法與架構 (How to do)」** 。它將 PRD 的需求轉化為具體的技術決策、資料流與模組劃分。
* **核心要素**：
  * **架構與約束條件 (Constraints)**：例如限制只能使用 `requests` 與 `beautifulsoup4`，不可使用付費 API。這能防止 AI 自作主張引入不必要的龐大依賴套件。
  * **執行流程 (Execution Flow)**：利用流程圖（如 Mermaid 語法）規範從 CLI 參數解析、路由判斷到資料輸出的完整路徑。
  * **風險評估 (Risks)**：預判系統潛在的脆弱點。例如網頁 HTML 結構若改變，爬蟲就會失效，這在 SDD 中必須被標記為已知風險。

## 二、 滾雪球開發策略 (Snowball / Iterative Development)

教授在課堂中特別強調，面對複雜系統，絕對不能要求 AI「一次寫完所有功能」。必須採取**分階段的迭代策略 (Iterative Phases)**，這就是規格驅動開發 (SDD) 中的「滾雪球」。

### 開發階段拆解示範 (以 CLI 系統為例)

* **Phase 1: 基礎驗證 (Minimum Viable Prototype)**
  * **目標**：只做最核心的連線與抓取。驗證爬蟲邏輯是否能正確取得 Google Finance 的 HTML，並解析出價格。不處理任何資料庫或複雜演算法。


* **Phase 2: 本地資料庫整合**
  * **目標**：加入 `--his_insert` 參數，實作 SQLite 歷史資料寫入。在此階段，我們將 Phase 1 抓到的資料落地，並確保 Schema 設計正確。


* **Phase 3 & 4: 擴充查詢與演算法分析**
  * **目標**：加入單一股票查詢 `--ticker`，並實作 `--avg_break`（計算移動平均線 MA5/10/20 並偵測突破）。


> **💡 教授重點補充**：每一階段都必須建立在「前一階段已測試通過」的基礎上。如果在 Phase 4 發現錯誤，範圍就能限縮在移動平均線的計算邏輯，而不需要懷疑是 Phase 1 的爬蟲壞掉。

## 三、 現實生活中的應用實例與延伸思考

這部分結合了課程中的 PRD/SDD 規範，以及高階軟體工程與資安領域的實務場景：

**1. 醫療器材軟體 (SaMD) 的 FDA 審查與文件追溯**

* **實例**：在開發如「DeepBT Detector-Plus」這類作為醫療器材的軟體 (SaMD) 時，PRD 與 SDD 不僅僅是開發工具，更是法規合規（如 FDA 510(k) 申請）的必要文件。FDA 審查員會嚴格檢視 PRD 中的「預期用途 (Intended Use)」是否與 SDD 中的「資安防護機制 (Cybersecurity Plan)」與「風險評估」完全對應。任何未在 SDD 中定義的 AI 幻覺代碼，都會導致醫療器材的安全認證失敗。

**2. 警政防詐系統 (如 165 專案) 的階段性開發**

* **實例**：在建構大型的語音分析防詐系統時，滾雪球策略至關重要。Phase 1 可能僅是確保 ASR (如 Whisper 模型) 能穩定且準確地將報案錄音轉為文字（處理雜音、方言）；Phase 2 才是導入 LLM 進行實體萃取與詐騙樣態分析；Phase 3 則是與現有的警政資料庫進行整合。如果一開始就要求 AI 把語音轉文字並同時輸出犯罪分析報告，系統極易崩潰且無法除錯。

**3. 高效能地端應用的非功能需求 (NFR)**

* **實例**：在開發即時語音轉譯系統 (如 EchoNote) 時，PRD 的「非功能需求」會是決定成敗的關鍵。例如規定「系統必須在 Ubuntu 環境下離線運行」、「延遲必須小於 0.5 秒」、「需充分利用 RTX 5080 等 GPU 加速」。有了這些 NFR 約束，在撰寫 SDD 時，開發者就能精準指示 AI 使用 Rust 語言結合 PyQt5 來構建系統，並採用 `faster-whisper` 來滿足低延遲的記憶體安全要求，而不是生成一個依賴雲端 API 的 Python 慢速腳本。

**4. 系統原型建構與介面契約 (API Contract)**

* **實例**：在快速打造原型系統（例如使用 Streamlit 作為前端，FastAPI 搭配 FAISS 向量資料庫作為後端）時，SDD 的價值在於定義前後端通訊的「契約」。即便後端檢索邏輯未來要抽換，只要 API 的輸入輸出格式（JSON Schema）在 SDD 中被嚴格鎖定，前端介面就不會受到影響。

# 📖 課程講義：AI 輔助高效能底層架構規格書 (SDD) 撰寫實務

## 一、 核心概念定義：底層架構的規格約束 (Constraints)

在使用 AI 撰寫底層架構 SDD 時，最重要的是定義**非功能需求 (NFR, Non-Functional Requirements)** 與**硬體約束**。缺乏這些，AI 就無法進行「降維打擊」。

### 1. 硬體拓撲與資源邊界 (Hardware Topology & Resource Boundaries)

* **定義**：明確定義系統運作的實體環境限制，包含 CPU 架構、GPU 記憶體 (VRAM) 大小、磁碟 I/O 速度等。
* **舉例**：在 SDD 中明確標註「目標執行環境為 Ubuntu 系統，需利用本機單張 NVIDIA RTX 5080 (16GB VRAM) 進行邊緣運算推論，且主記憶體佔用不得超過 4GB」。
* **AI 提示技巧**：這能強制 AI 在設計模型載入策略時，選擇量化 (Quantization) 技術或記憶體映射 (Memory Mapping)，而非將所有資料一次載入 RAM。

### 2. 併發與執行緒模型 (Concurrency & Threading Model)

* **定義**：規範系統如何同時處理多個任務而不互相阻塞 (Blocking)。在底層開發中，錯誤的併發設計會導致死結 (Deadlock) 或資源競爭 (Race Condition)。
* **舉例**：定義「音訊擷取 (Audio Capture)」、「神經網路推論 (NN Inference)」與「圖形使用者介面 (GUI)」必須運行在獨立的執行緒或非同步任務中，並透過無鎖佇列 (Lock-free Queue) 或通道 (Channel) 傳遞資料。

### 3. 記憶體安全與生命週期 (Memory Safety & Lifecycle)

* **定義**：規範系統如何配置與釋放記憶體。
* **舉例**：要求 AI 在設計介面 (API) 時，必須明確指出指標的擁有權 (Ownership) 歸屬，避免記憶體洩漏 (Memory Leak) 或懸掛指標 (Dangling Pointers)。

## 二、 撰寫高效能 SDD 的 AI 協作框架 (Constraint-First Prompting)

要讓 AI 產出高品質的底層 SDD，我們必須採用 **「約束優先 (Constraint-First)」**的提示策略。

1. **定義技術堆疊的「白名單」與「黑名單」**：
* *錯誤示範*：「幫我設計一個即時語音轉文字系統。」(AI 可能會給你一個呼叫 OpenAI API 的慢速 Python 腳本)
* *正確示範*：
「撰寫即時語音轉文字系統的 SDD。
  * **[白名單]**：核心邏輯使用 Rust 開發以確保記憶體安全與高效能；推論引擎使用 Faster-Whisper；前端介面使用 PyQt5。
  * **[黑名單]**：嚴禁使用任何外部雲端 API，必須完全地端 (On-premise) 執行。」


2. **要求 AI 定義外部函式介面 (FFI, Foreign Function Interface)**：
* 當底層高效能語言（如 Rust/C++）需要與其他生態系（如 Python 的 AI 生態圈或 PyQt GUI）溝通時，要求 AI 在 SDD 中詳細定義 FFI 的資料傳遞格式（例如：將字串轉為 C-style pointer 再傳遞）。

## 三、 現實生活中的應用實例

**1. 地端即時轉譯與會議摘要系統 (On-premise Real-time Transcription)**

* **實例**：在醫院診間或警政偵訊等具備高度機敏性的場域，語音資料無法上傳雲端。開發此類系統（例如以 Faster-Whisper 模型為基礎的即時應用）時，效能與資安是首要考量。
* **SDD 應用**：在 SDD 中，我們利用 AI 協助設計一個 **「生產者-消費者 (Producer-Consumer) 架構」**。
  * *生產者*：PyQt5 GUI 介面中的音訊錄製執行緒，負責將麥克風收音切片存入緩衝區。
  * *消費者*：後端的 Rust 異步任務 (如 Tokio framework)，負責從緩衝區取出音訊，調用 GPU 資源進行 ASR 推論。
  * SDD 必須規範兩者之間的 Channel 容量限制（避免 OOM）以及錯誤處理機制（如推論超時的應對）。



**2. 網路封包深度檢測系統 (Deep Packet Inspection, DPI)**

* **實例**：在建構 AI 實驗室的防火牆或進行滲透測試防護時，系統需要以線速 (Line-rate) 處理網路封包。
* **SDD 應用**：在撰寫此類系統的規格時，我們會要求 AI 在 SDD 中加入「零拷貝 (Zero-copy)」的網路 I/O 設計規範。這意味著封包從網卡驅動程式進入應用層時，不會在記憶體中被來回複製，而是直接透過指標傳遞給分析引擎，藉此榨出硬體的極限效能。

## 四、 延伸教材補充：高效能系統的架構驗證 (Architecture Validation)

當 AI 幫您產出底層架構的 SDD 後，您該如何驗證這個架構是可行的？

### 1. 概念驗證 (PoC, Proof of Concept) 的降維實作

在確認完整 SDD 之前，先請 AI 針對架構中 **「最高風險的技術節點」** 寫一個最小可執行代碼。

* **延伸操作**：如果 SDD 中設計了 Rust 與 Python 透過 FFI 互相呼叫，不要急著開發完整功能。先請 AI 寫一段「Rust 傳遞一個含有 10 萬個浮點數的陣列給 Python 並成功印出長度」的測試程式碼。如果這個 PoC 效能達標且不崩潰，才將該設計正式定案寫入 SDD。

### 2. 威脅建模 (Threat Modeling) 融入底層 SDD

高效能系統往往與底層硬體或作業系統核心 (Kernel) 互動密切，這也意味著較大的攻擊面。

* **延伸操作**：在使用 AI 撰寫 SDD 時，加入一個 `Security Constraints` 區塊。例如，要求 AI 評估該 Rust 應用程式在編譯時是否應開啟特定安全旗標，或者在 Linux 環境下執行時，是否能透過 `AppArmor` 或 `seccomp` 限制其 System Call 的權限，以防禦潛在的緩衝區溢位或提權攻擊。

# 📖 課程講義：底層記憶體管理的陷阱 — Memory Leak 與 Dangling Pointers

在 C/C++ 等允許手動管理記憶體的語言中，開發者擁有極大的彈性與效能優勢，但這也意味著必須對記憶體的「生命週期」負起全責。當管理失當，就會引發以下兩大嚴重問題。

## 一、 核心概念定義：記憶體洩漏 (Memory Leak)

### 1. 專有名詞定義

**記憶體洩漏 (Memory Leak)** 是指程式在執行期間，向作業系統動態配置了記憶體（通常是在 Heap 區段），但在使用完畢後，**忘記釋放 (Deallocate)** 該記憶體，導致這塊記憶體空間無法被系統回收再利用。

### 2. 原理與舉例

* **運作機制**：就像是你去圖書館借了書（配置記憶體），但永遠不還回去。隨著時間推移，圖書館的書會被借光，導致其他人（或你的程式後續的操作）無書可借。
* **程式碼範例 (C語言)**：
```c
void do_something() {
    // 在 Heap 中配置了 100 bytes 的記憶體
    char *buffer = (char *)malloc(100); 

    // ... 執行某些操作 ...

    // 函數結束時，指標變數 buffer 被銷毀了，
    // 但它指向的那 100 bytes 記憶體並沒有被 free() 釋放。這就是 Memory Leak。
}

```



### 3. 現實生活中的應用/災難實例

* **長時間運作的伺服器 (Backend Servers)**：如果是網頁伺服器或資料庫引擎存在記憶體洩漏，剛啟動時運作正常，但隨著處理的請求越來越多，消耗的記憶體會像雪球般越滾越大，最終觸發作業系統的 **OOM (Out of Memory) Killer** 導致伺服器崩潰 (Crash)。這也就是為什麼有些設計不良的系統需要「每隔幾天設定排程自動重啟」的原因。
* **網頁瀏覽器 (Web Browsers)**：早期的瀏覽器分頁若存在記憶體洩漏，當你開啟大量分頁並關閉後，記憶體卻沒有歸還給系統，導致整台電腦變得極度卡頓。

---

## 二、 核心概念定義：懸掛指標 (Dangling Pointers)

### 1. 專有名詞定義

**懸掛指標 (Dangling Pointer)**（又稱迷途指標）是指一個指標變數，它所指向的記憶體位址已經被**釋放 (Freed)** 或**失效**了，但該指標本身卻沒有被重置為 `NULL`，仍然「懸掛」著指向那塊無效的記憶體。

### 2. 原理與舉例

* **運作機制**：就像是你擁有某個飯店房間的鑰匙（指標），但你已經退房了（記憶體被釋放）。後來這個房間被分配給了別人，如果你此時又拿著舊鑰匙去開門（對指標進行 Dereference 取值或寫入），就會闖入別人的房間，引發嚴重的混亂。
* **程式碼範例 (C語言)**：
```c
char *ptr = (char *)malloc(100); // 配置記憶體
// ... 使用 ptr ...

free(ptr); // 釋放了記憶體，退還給系統

// 此时 ptr 成為了「懸掛指標」
// 錯誤示範：繼續使用已經釋放的指標
strcpy(ptr, "Hack the system"); // 未定義行為 (Undefined Behavior)，可能導致崩潰！

```



### 3. 現實生活中的應用/災難實例

* **資安漏洞：釋放後使用 (Use-After-Free, UAF)**：在資訊安全領域，懸掛指標是駭客最愛的攻擊面之一。駭客可以透過精心設計的操作，讓系統釋放某塊記憶體（產生懸掛指標），接著駭客自己填入惡意程式碼到該記憶體位址。當原本的程式不知情地再次使用該懸掛指標時，就會執行到駭客的惡意程式碼，達成**任意程式碼執行 (Arbitrary Code Execution)**，進而控制整台主機。
* **遊戲引擎崩潰 (Game Engine Crashes)**：在複雜的 3D 遊戲引擎中，如果某個怪物物件已經被玩家擊殺並從記憶體中刪除，但尋路系統的指標依然指向該怪物。當尋路系統試圖讀取怪物的座標時，就會讀取到無效記憶體，導致遊戲閃退 (Segmentation Fault)。

---

## 三、 延伸教材補充：現代系統如何防禦這些陷阱？

為了對抗這些存在數十年的底層問題，現代軟體工程發展出了許多防禦機制：

**1. 記憶體檢測工具 (Memory Profilers)**

* **Valgrind**：Linux 系統下最經典的動態分析工具。它可以在程式執行時監控所有記憶體的配置與釋放，並在程式結束時精準列出哪一行程式碼造成了 Memory Leak。
* **AddressSanitizer (ASan)**：Google 開發的編譯器擴充功能（支援 GCC / Clang）。在編譯時加入 `-fsanitize=address` 參數，程式若在執行期觸發了懸掛指標 (Use-After-Free) 或記憶體越界，會立刻拋出詳細的錯誤報告並終止程式。

**2. 現代程式語言的機制**

* **垃圾回收機制 (Garbage Collection, GC)**：如 Java、Python、Go 等語言，由系統底層的 GC 背景程式自動追蹤哪些記憶體不再被使用並自動回收，幾乎消除了 Memory Leak 和 Dangling Pointers 的問題，但代價是會有效能損耗與不可預測的停頓 (GC Pause)。
* **C++ 的智慧指標 (Smart Pointers)**：現代 C++ (C++11 以後) 引入了 `std::unique_ptr` 和 `std::shared_ptr`。它們利用物件的生命週期（RAII 機制），當指標變數離開作用域時，會「自動」呼叫 `free/delete` 釋放記憶體，大幅降低人為疏失。
* **Rust 的所有權系統 (Ownership & Borrowing)**：Rust 語言在**不使用垃圾回收器**的前提下，透過編譯器嚴格規定「一塊記憶體同時間只能有一個擁有者」，且「借用（指標）的生命週期不能長於擁有者」。這從編譯階段就**物理性地消滅**了懸掛指標與二次釋放 (Double Free) 的可能性，也是現代高效能安全系統的首選語言。

# 📖 課程講義：進階系統架構與資料倉儲 (Zero-copy & Snowflake)

## 一、 系統底層效能的極致：零拷貝 (Zero-copy)

在高效能伺服器與網路程式設計中，資料的傳輸速度往往受限於 CPU 在不同記憶體區塊間「搬運」資料的過程。Zero-copy 就是為了解決這個效能瓶頸而生的技術。

### 1. 核心概念定義

**零拷貝 (Zero-copy)** 是一種作業系統級別的技術。它的字面意思並不是「完全不複製資料」，而是指 **「CPU 不需要耗費資源去執行將資料從一個記憶體區域複製到另一個記憶體區域的任務」**。透過減少應用程式（User Space）與作業系統核心（Kernel Space）之間的資料拷貝次數與內容切換 (Context Switch)，大幅提升系統吞吐量 (Throughput)。

### 2. 傳統資料傳輸 vs. 零拷貝

* **傳統傳輸模式**：假設你要將硬碟裡的一個檔案透過網路傳送出去，傳統流程需要經過 **4 次資料拷貝**與 **4 次內容切換**。
1. 硬碟 $\rightarrow$ 系統核心緩衝區 (Kernel Buffer)
2. 系統核心緩衝區 $\rightarrow$ 應用程式緩衝區 (User Buffer) —— *這步最浪費 CPU*
3. 應用程式緩衝區 $\rightarrow$ 網路 Socket 緩衝區
4. Socket 緩衝區 $\rightarrow$ 網路卡 (NIC)


* **零拷貝模式 (如 Linux 的 `sendfile` 指令)**：應用程式只發送傳輸指令，資料直接在系統核心層面完成轉移。
1. 硬碟 $\rightarrow$ 系統核心緩衝區
2. 系統核心緩衝區 $\rightarrow$ 網路卡 (NIC) —— *完全略過應用程式層，減少了 CPU 介入*



### 3. 現實生活中的應用實例

* **Apache Kafka (分散式訊息佇列)**：Kafka 之所以能在一台普通的伺服器上達成每秒數百萬則訊息的高吞吐量，最關鍵的底層技術就是 Zero-copy。當消費者 (Consumer) 來讀取訊息時，Kafka 直接指令作業系統將磁碟中的資料透過網路卡送出，資料完全不會進入 Kafka 應用程式的記憶體空間。
* **Nginx (高效能網頁伺服器)**：在傳輸靜態檔案（如圖片、影片、HTML）時，Nginx 預設會開啟 `sendfile` 零拷貝機制，讓檔案傳輸效能達到硬體的極限。

## 二、 資料倉儲架構：雪花模型 (Snowflake Schema)

在資料倉儲 (Data Warehouse) 領域，"Snowflake" 通常指的是一種資料庫的綱要 (Schema) 設計模式，用來組織關聯式資料庫中的龐大歷史數據。

### 1. 核心概念定義

**雪花模型 (Snowflake Schema)** 是一種多維度資料模型，它是「星狀模型 (Star Schema)」的延伸。在雪花模型中，中心的 **「事實表 (Fact Table)」** 連接著多個 **「維度表 (Dimension Tables)」**，而這些維度表會進一步被 **「規格化 (Normalized)」**，拆分成更多層級的子維度表。因為實體關聯圖 (ER Diagram) 展開後形似一片雪花，故得此名。

* **事實表 (Fact Table)**：儲存發生的事件或交易數據（例如：銷售金額、銷售數量、時間戳記）。
* **維度表 (Dimension Table)**：儲存描述事實的屬性（例如：客戶資料、產品類別、地區資訊）。
* **規格化 (Normalization)**：為了消除資料重複而將表格拆分的過程。

### 2. 星狀模型 vs. 雪花模型 (舉例說明)

假設我們有一家跨國電商的資料倉儲：

* **星狀模型**：中心是「訂單事實表」。旁邊有一個「商店維度表」，裡面同時包含了「店名、城市、州份、國家」。這會導致「國家」名稱被重複儲存無數次。
* **雪花模型**：為了節省空間與維持資料一致性，我們將「商店維度表」拆解。中心「訂單」連到「商店」表，「商店」表連到「城市」表，「城市」表再連到「國家」表。

### 3. 現實生活中的應用實例

* **企業級 BI (商業智慧) 報表系統**：當企業擁有極度龐大的維度資料，且高度重視資料的**一致性與更新效率**時（例如：如果國家名稱變更，雪花模型只需更新「國家表」的一筆紀錄，星狀模型則要更新數百萬筆），資料工程師會選擇採用雪花模型來建構資料超市 (Data Mart)。
* **缺點與權衡**：雖然雪花模型節省了儲存空間，但在查詢時需要執行大量的 `JOIN`（關聯）操作，會導致查詢效能下降。在儲存成本極低的現代，許多大數據系統反而傾向退回使用反規格化的「星狀模型」或「大寬表 (Wide Table)」來換取極致的查詢速度。

## 三、 延伸教材補充

**1. 系統底層補充：什麼是 DMA (Direct Memory Access)?**

* **定義**：在 Zero-copy 技術中，能讓資料從硬碟直接飛到系統核心緩衝區的幕後功臣叫做 **DMA（直接記憶體存取）** 控制器。它是一個獨立於 CPU 之外的硬體晶片。當需要讀寫大量資料時，CPU 會把任務交代給 DMA，由 DMA 負責搬運資料，CPU 就可以轉頭去處理其他高價值的運算邏輯。這完美體現了系統設計中「分工」的藝術。

**2. 業界術語補充：Snowflake 雲端資料平台**

* **定義**：在現代軟體工程中，當大家提到 "Snowflake" 時，除了指上述的資料結構外，有 90% 的機率是在談論一家名為 **Snowflake Inc.** 的雲端大數據公司。
* **技術突破**：傳統的資料庫，儲存 (Storage) 與運算 (Compute) 是綁在一起的。Snowflake 平台最大的革命在於 **「儲存與運算完全分離」**。這意味著企業可以把 PB 級的資料便宜地丟在 AWS S3 上，只有在業務單位需要跑複雜報表時，才動態開啟運算節點來分析資料，用完即關，大幅顛覆了現代資料工程的成本結構。
