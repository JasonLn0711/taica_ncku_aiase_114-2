# 📖 課程講義：AI 協作開發實務、架構文件與迭代策略 (Vibe Coding Part 3)

## 一、 避開 AI 開發陷阱：事前調研 (Research First)

AI 看似無所不能，但若開發者未經引導直接下達撰寫代碼的指令，AI 常會給出看似合理卻無法執行的方案，或踩中現實環境的地雷。

### 1. 核心概念：事前調研 (Pre-development Research)

* **定義**：在讓 AI 撰寫第一行程式碼之前，先將 AI 當作「技術顧問」。詢問它：「我想達成某個目標，目前業界有沒有人做過？推薦使用什麼開源套件、框架或架構？」
* **目的**：避免重複造輪子，並避開高風險的技術債。

### 2. 現實生活中的應用與陷阱實例

* **網路通訊協定的盲區 (IPv4 vs IPv6)**：課程中提到，AI 產生的連線代碼經常忽略了客戶端或伺服器端網路環境的差異。例如，在實驗室網路或高效能伺服器環境中，若 AI 預設綁定 IPv6，而底層網路基礎設施僅支援 IPv4，就會導致本地端測試正常，一部署就完全斷線的窘境。
* **資料爬蟲與授權風險**：以開發股價查詢系統為例，若直接叫 AI「去抓 Google Finance 的股價」，AI 會寫出使用 `BeautifulSoup` 硬爬網頁的代碼。這不僅會面臨網頁 HTML 結構隨時改變的維護風險，更可能違反服務條款 (TOS) 導致 IP 被封鎖，甚至面臨法律訴訟。透過事前調研，AI 會建議使用如 `yfinance` 這類合法且穩定的現成 API 函式庫。

## 二、 規格文件與版本控制 (Documentation & Version Control)

AI 產出的內容必須被標準化、結構化地保留下來，否則專案將淪為一堆散落的聊天紀錄。

### 1. Markdown (.md) 格式與 README

* **定義**：Markdown 是一種輕量級標記語言。課程要求所有 AI 產出的重要設定、環境需求與豐富格式 (Rich format)，都必須標準化地記錄在 `README.md` 或 `content.md` 中。
* **應用實例**：在多人協作或將專案交接給其他 AI Agent 時，一份格式標準的 Markdown 文件是唯一的溝通橋樑。

### 2. 版本管理 (Version Control)

* **定義**：利用 **Git** 與 **GitHub** 等工具來管理程式碼的變更歷史。* **實務規範**：課程強調，在任何規格文件或腳本的開頭（Header），必須明確標註版本號（如 `v1.0`, `v2.0`）。`v1.0` 之前通常是概念驗證 (Prototyping)；`v1.0` 代表具備基礎功能；`v2.0` 則通常代表經歷了底層架構的大幅重構。

## 三、 系統架構的視覺化溝通 (Visualizing Architecture)

不要用冗長且發散的文字向 AI 描述系統邏輯，人類與 AI 都需要「工程圖紙」。

### 1. Mermaid.js (Diagrams as Code)

* **定義**：一種基於 JavaScript 的繪圖工具，允許開發者使用類似 Markdown 的純文字語法來生成圖表。AI 非常擅長撰寫 Mermaid 語法。

### 2. UML 系統分析圖表

透過 Mermaid，我們可以輕易生成以下圖表來約束 AI 的架構設計：* **流程圖 (Flowchart)**：定義資料的走向與邏輯判斷的節點。

* **循序圖 (Sequence Diagram)**：定義不同模組（例如：前端介面、後端引擎、資料庫、外部 API）之間資料傳遞的「時間順序」與「生命週期」。
* **狀態圖 (State Diagram)**：描述系統物件在不同條件下的狀態轉換軌跡。
* **應用實例**：在設計自動化工作流（例如串接多個系統 API）或規劃資安防禦架構時，要求 AI 先產出循序圖。這能讓工程師在寫代碼前，一眼抓出 API 授權檢核是否有邏輯漏洞，或是資料流是否符合規範。

## 四、 滾雪球開發策略與可維護性 (Iterative Development)

### 1. 滾雪球式開發 (Iterative "Snowball" Development)

* **定義**：將大型專案拆解為多個微小的開發階段 (Phases)。第一步只求打通核心邏輯，驗證可行性，接著再逐步把功能「滾」大。* **實例**：先請 AI 寫一個只會印出「Hello World」和硬體參數的基礎外殼；測試通過後，再加入資料庫讀取模組；最後才加入複雜的演算法。

### 2. 「過早的最佳化是萬惡的根源」 (Premature optimization is the root of all evil)

* **概念**：這是軟體工程的經典名言。不要企圖叫 AI 一次寫出包含完整錯誤處理、非同步多執行緒且效能極致的 3000 行程式碼。
* **風險**：如果一開始就把架構寫得太複雜，一旦程式崩潰 (Crash)，你將完全無法定位問題在哪裡。

### 3. 可維護性 (Maintainability) 與程式碼審查 (Code Review)

* **教授重點強調**：**「如果你看不懂 AI 寫的 Code，你就無法 Maintain 它。」**
* **應用實例**：在開發需要極高穩定性的底層應用（例如使用具備嚴格編譯器檢查的語言如 Rust）時，如果 AI 產生了一段過度複雜、充滿黑魔法的指標操作，即便當下編譯能過，也絕對不能採用。無法被人類進行程式碼審查 (Un-reviewable) 的系統，對企業與組織而言就是隨時會引爆的技術債。

## 五、 延伸教材補充

**1. 網路效應與 AI 開源生態 (AI-SEO)**

* **補充**：錄音中短暫提到了類似 SEO（搜尋引擎最佳化）的網路效應概念。在 AI 時代，越早被業界採用、在 GitHub 上擁有越多星星 (Stars) 與討論度的開源工具，會在 LLM 的訓練資料中佔據更大的權重。這意味著，選擇主流的架構與技術堆疊，AI 就能給出更精準、更少幻覺的輔助；反之，若選擇極度冷門的框架，AI 的表現就會大幅下降。

**2. 自動化架構師 (The Automated Architect Agent)**

* **補充**：結合上述所有概念，現代高階開發流程中會引入一個「雙 Agent 環路」。第一個 Agent 扮演「系統架構師」，它的唯一工作是與您對話，並將對話內容轉化為標準的 Markdown 規格書與 Mermaid 系統架構圖。待您人工確認這份「圖紙」無誤後，再將這份文件餵給第二個扮演「工程師」的 Agent 進行滾雪球式的代碼撰寫。這種方法能最大程度地收斂 AI 的發散性。
